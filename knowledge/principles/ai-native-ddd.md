# AI-Native DDD

## 目的

この文書は、agent が DDD を「抽象的な設計論」ではなく、実務で使う判断基準として扱うための短いメモである。

## 基本姿勢

- まずコードの配置より、業務上の意味の境界を見る
- entity / value object / use case という名前より、「どこに判断を置くと壊れにくいか」を優先する
- agent は賢さで横断実装しがちなので、境界を意識して自制する

## 判断の置き場所

### domain に置くもの

- 業務ルール
- 状態遷移の制約
- その概念が常に守るべき不変条件

例:

- ある状態から別の状態へ進めてよい条件
- 名前や識別子の整合
- 集約単位で守るべき整合性

### application / use case に置くもの

- 1 つの操作として何を呼ぶか
- どの repository や service をどう組み合わせるか
- 権限確認、入出力整形、実行順序

### infrastructure に置くもの

- DB、HTTP、外部 API、ファイル I/O
- 永続化都合の変換
- ライブラリ依存の実装詳細

## AI 時代に特に重要なこと

### 1. 用語を固定する

- 同じ概念に複数の呼び名を使わない
- Task と Job、Story と Epic のような混線を避ける
- PR、コメント、文書、コードで同じ言葉を使う

### 2. 不変条件を先に言語化する

- agent はコードを足すのは得意だが、暗黙ルールを壊しやすい
- 「何が常に成り立つべきか」を先に書く
- 不変条件が書けないなら、まだ domain が見えていない

### 3. 便利な横断実装を警戒する

- 1 か所で早く直せる実装でも、複数の文脈を混ぜるなら長期的には危ない
- UI 都合の分岐を domain に入れない
- DB の都合で業務ルールを曲げない

### 4. 大きい変更ほど、先に境界を固定する

- 変更前に「どの境界は触るか」「どこは触らないか」を決める
- それが決まらない変更は、いったん Story を分ける

## 実務チェック

- この判断は業務ルールか、単なる手順か
- この制約は domain で守るべきか、呼び出し側で守ればよいか
- その命名は他の agent が見ても同じ意味で読めるか
- いま追加しようとしている分岐は、別の文脈の事情を持ち込んでいないか

## 避けること

- DDD の用語だけ増やして判断基準がない状態
- entity を薄くして、重要なルールを use case に散らすこと
- 逆に何でも domain に押し込んで境界を失うこと
