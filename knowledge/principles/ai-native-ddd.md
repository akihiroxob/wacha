# AI Native DDD

## 目的

この文書は、AI（エージェント）を前提とした開発において、
Domain-Driven Design（DDD）と Clean Architecture をどのように解釈し、
どのように適用するかの指針を定義する。

従来のDDDをそのまま適用するのではなく、
「AIが継続的に開発・理解・改善できる構造」を作ることを目的とする。

---

## 基本思想

### 1. Artifact中心設計

AIにとっての最小理解単位は「コード」ではなく「Artifact」である。

- Task
- Story
- Code
- Document
- Decision

これらを中心に設計し、変更履歴と判断理由が追跡できる状態を作る。

---

### 2. 責務による分離

DDDの本質は「責務の分離」であり、
クラスやファイルを増やすことではない。

主な責務：

- domain: ビジネスルール
- application: ユースケース
- infrastructure: 外部技術
- presentation: 入出力
- mcp: エージェント操作

---

### 3. AIが理解できる構造

構造は「理論的に正しい」よりも、
「AIが一貫して理解できる」ことを優先する。

- 命名が一貫している
- 配置が予測可能
- 責務が明確
- 変更理由が追跡できる

---

## 実務上の優先順位（重要）

このドキュメントは設計原則を示すものであり、
すべての変更で完全なレイヤ分離や構造再設計を要求するものではない。

実装時の優先順位は以下とする：

1. 既存構造との整合性
2. 最小差分での変更
3. 変更容易性・テスト容易性
4. 必要な範囲での責務分離
5. DDD / Clean Architecture の適用

---

## 実務ルール

### 小さい変更では構造を増やさない

- Entity / Repository / Service を一括で新規作成しない
- 新しい層を安易に追加しない
- 抽象化は必要になるまで行わない

---

### 既存構造を優先する

- 新しい構造を作る前に、既存の類似実装を探す
- 命名・配置・責務は既存コードに合わせる
- 「正しさ」より「一貫性」を優先する

---

### 過剰設計を防ぐ

以下は禁止とする：

- 「DDDだから」という理由でのファイル増加
- 将来のためだけの抽象化
- 使用されていない層の追加
- 1 Task の範囲を超えた構造変更

---

### 境界は必要なときだけ引く

レイヤ分割は以下の場合にのみ行う：

- 変更理由が明確に分離できる
- テスト容易性が向上する
- 再利用性が明確に向上する

---

### Domainの純度を守る

- domain は外部技術に依存しない
- DB / API / フレームワークを持ち込まない
- ただし、小規模変更では無理に分離しない

---

## Artifactとの関係

DDDの各要素は、Artifactとして扱う：

- Entity → Domain Artifact
- UseCase → Application Artifact
- Repository → Infrastructure Artifact
- Controller → Presentation Artifact

これにより：

- 変更履歴が追跡可能
- 判断理由が記録される
- エージェント間で共有可能

---

## 避けること

- 理想構造を目指して既存コードを壊す
- 一貫性より設計の美しさを優先する
- 小さな変更で大規模リファクタを行う
- エージェントの判断に依存した構造変更

---

## まとめ

DDDは「理想構造を作るためのもの」ではない。

DDDは：

- 変更を局所化するための手段
- 責務を明確にするための手段
- AIが理解しやすくするための手段

である。

構造はシンプルに保ち、
必要なときにだけ進化させる。
