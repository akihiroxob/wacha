# Development Principles

## 目的

この文書は、Wacha を使って開発する agent が日々の実務判断に使う最小の原則をまとめる。

Wacha 自体の内部設計や将来実装の説明はここには書かない。

## 大前提

開発では、Clean Architecture と Domain-Driven Design の考え方を基盤にする。

ただし、それらはファイル数・抽象・レイヤを増やすためではない。

目的は次を高めることである。

- 変更容易性
- 再利用性
- テスト容易性
- 既存構成との整合性
- 後続 agent が理解できる一貫性

## 原則

### 1. 先に管理可能な単位へ落とす

- 依頼をそのまま実装に持ち込まない
- まず Story または Task に落とし、目的と完了条件を明確にする
- 1 回の review で判断できない単位は大きすぎる

### 2. 完了条件が書けない作業は、まだ着手しない

- 「何をもって終わりか」が書けないなら、実装より先に要件整理が必要
- 完了条件は挙動ベースで書く
- 「調査する」「いい感じに直す」だけでは完了判定できない

### 3. Story は意図、Task は実行

- Story には背景、狙い、完了像を書く
- Task には作業単位と確認可能な結果を書く
- Story に実装手順を詰め込みすぎない
- Task に背景の長文を持ち込みすぎない

### 4. 変更は小さく、境界は明確にする

- 1 Task で複数の責務をまたがない
- UI、ユースケース、永続化、運用文書を同時に変えるなら、本当に 1 つの完了条件で閉じるかを確認する
- 境界が曖昧なら先に分割する

### 5. 既存構成を最優先する

- 新しい構成を作る前に、既存の類似ファイル・類似ディレクトリを探す
- 既存の命名、責務分割、配置に合わせる
- 構成に迷ったら、新しいディレクトリを作らず、既存の近い場所に置く
- 判断できない場合は、推測で進めず task comment に前提と懸念を残す

### 6. 新規ファイル作成を抑制する

- 既存ファイルの編集で済むなら、新規ファイルを作らない
- 新規ファイルを作る場合は、配置理由と既存ファイルで代替できない理由を説明する
- 1 Task で新規ファイルが 3 つ以上必要な場合は、実装前に FileChangePlan を共有する
- 空のディレクトリや将来用のプレースホルダは作らない

### 7. Clean Architecture / DDD を過剰適用しない

- DDD 用語を増やすこと自体を目的にしない
- 小さい変更で Entity、UseCase、Repository、Service を一括追加しない
- 抽象化は、実際に変更理由を分離できる場合にだけ行う
- レイヤ分割は、変更容易性・再利用性・テスト容易性に寄与する場合にだけ採用する

### 8. SOLID は現実的に使う

- 単一責務を優先する
- 依存方向を守る
- 外部技術への依存を domain に持ち込まない
- ただし、過剰な interface や抽象クラスを増やさない
- 将来使うかもしれない拡張点は、今回の task では作らない

### 9. 既存挙動を守る責任を持つ

- 新機能でも、壊してはいけない既存挙動を明記する
- 確認方法がない変更を完了扱いにしない
- テストが重すぎる場合でも、最低限の確認手順は残す

### 10. 不明点は実装で埋めない

- 要件不明を推測で補わない
- 実装者判断で仕様を増やさない
- 迷ったら、前提を Task やコメントに言語化して返す

### 11. 後工程に優しい形で残す

- reviewer が見て意図を再構成しなくてよい状態にする
- reject されても次の担当者が困らない文章を書く
- ローカルな文脈ではなく、成果物から判断できる形を優先する

## FileChangePlan の原則

実装前に、最低限次を考える。

- 何を達成するか
- どの既存ファイルを編集するか
- 新規ファイルが必要か
- 新規ファイルが必要な場合、なぜ既存ファイルでは不十分か
- どの範囲は触らないか
- どう検証するか

## 避けること

- 大きい依頼を 1 Task に押し込む
- 実装と調査と要件整理を 1 つの完了扱いにする
- 変更理由が書かれていないまま review に出す
- 将来案や未確定案を、今回の完了条件に混ぜる
- 既存構成を確認せずに新規ディレクトリを作る
- Clean Architecture / DDD / SOLID を理由に、不要なファイルや抽象を増やす
