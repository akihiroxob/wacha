# Task Writing Tips

## 目的

Task を、worker が着手時に迷わず、reviewer が完了判定しやすい形で書く。

## 良い Task の条件

- 何を変えるかが 1 文で分かる
- 完了判定ができる
- 範囲外が分かる
- 必要な前提が書かれている

## まず決めること

- この Task は何を終わらせるのか
- どの挙動を確認できれば完了か
- 壊してはいけない既存挙動は何か
- 不明点があるなら、実装前に誰が判断するか

## 書き方

Task の description は Gherkin を基本にする。

```md
Given 前提
When 実行すること
Then 確認したい結果
And 補足の結果や非対象
```

## 書くべき内容

- 対象: どの画面、API、ユースケース、文書か
- 結果: 何が変わるか
- 制約: 何は変えないか
- 確認観点: テストまたは手動確認で何を見るか

## そのまま着手できる Task の目安

- worker が追加質問なしで最初の一手を決められる
- reviewer が reject 条件を想像できる
- 「終わったかどうか」をコード以外の言葉で説明できる

## 書き直したほうがよい Task

### 悪い例

```md
Task をいい感じに整理する
```

何をもって完了か分からない。

### 改善例

```md
Given Project 画面で Task 一覧を表示している
When status ごとの件数を集計して表示する
Then `wait_accept` を含む全状態の件数が見える
And 既存の `todo` `doing` `in_review` `accepted` `rejected` 表示を壊さない
```

## 分割したほうがよい兆候

- 「調査して決める」が主文になっている
- 2 つ以上の成果物に別々の完了条件がある
- reviewer が複数の論点を別々に判断しないと通せない
- 実装と follow-up 設計が同居している

## reject されにくい書き方

- Then は観測可能な結果にする
- And で既存維持条件を書く
- 非対象を短く書く
- 曖昧語を減らす

避けたい曖昧語:

- 適切に
- 必要に応じて
- 可能なら
- いい感じに
- 柔軟に

## 最低限のテンプレート

```md
Given 対象と前提が分かっている
When worker が変更を行う
Then 期待する結果が確認できる
And 壊してはいけない条件が維持される
```
