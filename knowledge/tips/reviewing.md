# Reviewing Tips

## 目的

review を「感想」ではなく、「通してよいかの判定」として行う。

## 最初に見ること

- Task の `Then` と `And` を満たしているか
- 既存挙動を壊していないか
- 変更理由と変更範囲が読めるか
- 確認方法が残っているか

## review の優先順位

### 1. Correctness

- 指示どおりに動くか
- 状態遷移や条件分岐に破綻がないか

### 2. Regression Risk

- 既存ユーザーが困る壊れ方をしていないか
- 周辺ケースの抜けがないか

### 3. Completeness

- Task に書かれた完了条件が全部満たされているか
- 一部だけできていて完了扱いになっていないか

### 4. Clarity

- 次の担当者が意図を読めるか
- 命名やコメントが誤解を増やしていないか

## その場で直してよいもの

- typo
- 表記ゆれ
- 自明なコメント補足
- 既存仕様を固定するだけの小さなテスト不足

## 差し戻すべきもの

- ロジック変更が必要
- 要件解釈で正解が変わる
- テストや確認が足りず安全に通せない
- 直す範囲が複数責務に広がる
- reviewer が実装者にならないと終わらない

## reject reason の書き方

最低限、次を含める。

- 何が不足しているか
- なぜ危ないか
- 何を満たせば再レビュー可能か

悪い例:

- なんとなく不安
- 直してください

良い例:

- `Then` にある状態遷移は実装されているが、`And` にある既存一覧表示の維持を確認するテストがない。回帰有無を判断できないため、一覧表示を固定する確認を追加して再レビューに出してほしい。

## 通してはいけない状態

- 未確認のまま「たぶん大丈夫」で終える
- 追加対応が必要なのに完了扱いにする
- 重要な不明点を comment に埋めて status だけ進める

## review で見るべきでないもの

- 自分の好みだけの書き方差
- 将来やるかもしれない改善を今回の必須条件にすること
- Task にない別テーマを混ぜて差し戻すこと

## 最後の問い

この Task を通したあとで問題が出たとき、reviewer は「見えていたのに流した」と言われないか。

その答えが yes なら、通さない。
